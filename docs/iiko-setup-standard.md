## 1. Дерево номенклатуры (критично)

В iikoOffice все номенклатурные группы собраны под три корневые группы
с каноническими именами:

```
Кухня/     ← всё, что готовит кухня (горячее, салаты, супы, десерты, ...)
Бар/       ← напитки и крепкий алкоголь (кофе, лимонады, коктейли, виски, ...)
Вино/      ← всё вино (бокалы, бутылки, игристые, вермуты)
```

Дашборд читает группу 1-го уровня (`DishGroup.TopParent`) и не содержит
никаких списков категорий.

**Правило атрибуции (единственное, зашитое в код):**

- группа лежит внутри одной из трёх папок → все её продажи относятся
  к этому юниту («Прочее» внутри «Бар» = бар);
- группа вынесена в корень (вне трёх папок) → её продажи **не входят
  в разрезы по юнитам** (донат, фудкост юнитов), но **входят в общую
  выручку** и итоговые показатели.

Ресторан сам решает, что положить в папки, а что держать вне дерева —
дашборд просто отражает это решение. Единственное жёсткое требование:
три корневые папки существуют и содержат основное меню.

**Внутренняя структура папок — свободная.** Дашборд читает только
1-й уровень; сколько подгрупп и какой вложенности внутри «Кухни» —
не регламентируется, в донат юнитов они не попадают (подгруппы видны
на втором уровне детализации страницы продаж).

**Самокоррекция:** продажи вне трёх папок дашборд показывает как
«вне подразделений» — если ресторан видит там заметную сумму, значит,
какая-то группа лежит не в папке; поправить дерево можно в любой момент,
изменения подхватываются со следующей выгрузки.

**Статус пилота (июль 2026): НЕ выполнено.** 37 групп лежат в корне плоским

## 2. Типы заказов (критично)

Заведён фиксированный набор типов заказов (и только он):

| Тип | Назначение |
|---|---|
| Обычный заказ | зал |
| Конференции | банкеты и мероприятия |
| Бар | барные чеки |
| Завтраки | завтраки отеля (если применимо) |

Персонал обязан выставлять тип при открытии заказа (см. onboarding-iiko.md,
часть 1). Дашборд использует значения как есть; новый тип, не согласованный
с разработкой, разрежет аналитику.

## 3. Безвыручные категории (критично)

Настроены механизмы с каноническими именами:

- скидка **«Комплимент»** 100% — угощения гостям;
- скидка или тип оплаты **«Представительские»** — расходы заведения;
- счета списания **«Проработка»** и **«Питание персонала»** — для актов
  списания (НЕ через кассу).

Без них потери размазываются по скидкам и «грязный фудкост» не считается.

## 4. API-доступ (критично)

- Отдельный API-пользователь только для дашборда;
- права: чтение отчётов (OLAP), без прав на кассовые операции и справочники;
- стойкий пароль (не 4 символа), хранится только в `.env` контейнера;
- после внедрения пароль ротируется, если передавался по открытым каналам.

## 5. Гигиена справочников

- Никаких категорий-опечаток (на пилоте найдена категория `,fh` — «бар»
  в английской раскладке) и групп-помоек (группа «Удаленные» с продажами);
- переименование типов оплат/заказов/групп — только с уведомлением
  разработки ДО изменения;
- хвостовые пробелы в названиях блюд дашборд стрижёт сам, но лучше их
  не плодить.

## 6. Валюта и локаль инстанса

Валюта ресторана задаётся в конфигурации контейнера (`.env`), фронтенд
подписывает суммы по ней. В коде валюта не хардкодится.

## Чек-лист аудита при внедрении

Быстрая проверка кассы кандидата (по OLAP за последний закрытый месяц):

- [ ] `DishGroup.TopParent` возвращает только Кухня/Бар/Вино
- [ ] типы заказов — из стандартного набора, доля «Обычный заказ»
      у банкетных сумм < 5%
- [ ] есть скидка «Комплимент» и счета «Проработка»/«Питание персонала»
- [ ] доля удалённых заказов и блюд в норме (< 3–5% выручки; на пилоте
      до внедрения было 37% фантомной выручки от удалённых)
- [ ] нет категорий-опечаток и групп вне дерева
- [ ] API-пользователь заведён, доступ проверен

План: оформить аудит скриптом (подключение к iiko кандидата -> отчёт
о соответствии стандарту за 5 минут) — он же продажный инструмент внедрения.
