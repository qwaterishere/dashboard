## 1. Исключение позиций из продаж и себестоимости (стафф и подобные)

**Статус:** дизайн утверждён (вариант «А», 09.07.2026), ждёт реализации.

**Проблема.** Рестораны проводят через кассу не-гостевые продажи —
питание персонала по внутренней цене (реальный кейс второго ресторана:
группа «Кофе, чай Staff», позиции по 700 сом). Такие строки загрязняют
гостевую аналитику: выручку, средний чек, qty позиций, фудкост.

**Утверждённое решение (вариант А):** стафф-позиции полностью
исключаются из гостевой аналитики, но остаются в данных (флаг в БД)
для будущего отчёта «внутренние обороты» (вместе с комплиментами
paid_total = 0).

**Механика:**

- `STAFF_MARKERS = ('staff', 'стафф', 'персонал', 'служебн')` +
  функция `is_staff(*names)` — в domain-константы, рядом с `resolve_unit`,
  та же терпимость к регистру. Маркер ищется в именах ГРУППЫ, КАТЕГОРИИ,
  папки 1-го уровня и ТИПА СКИДКИ (кейс кофейни: скидка «STAFF»
  на обычные позиции, 61 тыс. — группой не ловится; поле из №6) —
  НЕ в имени блюда (ложные срабатывания). Стандарт при этом требует
  переводить staff-скидки на акты — детектор нужен на переходный период;
- маркировка на этапе ingest: колонка `DishSale.is_staff`
  (материализация как у `paid_total`; `replace_day` переклассифицирует
  при перезагрузке дня);
- `Order.paid_total` считается БЕЗ стафф-строк → чек целиком из стаффа
  автоматически выпадает из чеков/гостей/среднего по уже существующему
  правилу «paid_total > 0 = продажа» (дашборд-сервис не трогается);
- `build_sales`: фильтр строк `NOT is_staff` (qty и цены — только гостевые);
- `_unit_sums` (фудкост юнитов): стафф-строки исключить из cost/revenue;
- аудит (`check_directory_hygiene`): staff-маркеры в продажах →
  предупреждение «переведите питание персонала на акты списания»
  (стандарт, раздел 3);
- документы: маркеры — в паспорт внедрения (канон для всех ресторанов);
  оговорка о сверке (выручка дашборда = гостевая, расходится с валовой
  кассой на сумму стаффа); handoff — семантика выручки;
- тесты: платный стафф-чек не в метриках; смешанный чек (гостевые
  строки остаются); ресторан без стаффа — no-op;
- перезаливка активной БД (новая колонка; create_all не делает ALTER).

**Проверка на живых данных:** перезалить кофейню, сверить «до/после» —
выручка должна упасть ровно на сумму staff-оборотов.

## 2. Не загружать нулевые модификаторы (техшум кассы)

**Статус:** дизайн согласован (10.07.2026), ждёт реализации
(удобно делать одним пакетом с №1 — общая перезаливка).

**Проблема.** Модификаторы без цены и себестоимости («Прожарка» —
степень прожарки стейка, «молоко на выбор» с нулевым прайсом,
«Листовой Чай» и т.п.) попадают в выгрузку как строки продаж.
Они нигде не считаются (фильтры отсекают), но раздувают БД
(у кофейни ~10% таблицы dish_sales) и шумят в аудите гигиены.

**Решение:** не записывать в БД строки с «тройным нулём».

**Критерий (важно — все три условия):**
`price == 0 AND paid_sum == 0 AND (cost IS NULL OR cost == 0)`.
Одного «продано за 0» НЕДОСТАТОЧНО: комплимент (скидка 100%) тоже
paid_sum = 0, но price > 0 — он НЕСЁТ смысл (гостеприимство/потери)
и обязан остаться. Платные модификаторы (молоко за деньги) не подпадают.

**Механика:**
- фильтр в начале `ingest_records` (до схлопывания сплитов);
- тесты: тройной ноль не в БД; комплимент (price>0, paid=0) остаётся;
  платный модификатор остаётся;
- аудит гигиены: нулевые строки перестают попадать в отчёт сами собой;
- паспорт: оговорка «нулевые модификаторы не загружаются; статистика
  по ним (степени прожарки и т.п.) — из iiko напрямую при need»;
- перезаливка активных БД.

**Осознанная цена:** в реплике исчезает след модификаторов
(«сколько well done») — при необходимости данные есть в iiko.

## 3. Фудкост только по строкам с себестоимостью (cost > 0)

**Статус:** правило утверждено (10.07.2026), ждёт реализации.

**Правило (система учёта строк):**
- продажная аналитика (выручка, qty, чеки, средний) — строки `paid_sum > 0`;
- аналитика фудкоста — строки `paid_sum > 0 И cost > 0`
  (ни в числителе, ни в знаменателе строкам без техкарт не место).

**Чинит две реальные проблемы:**
1. бизнес-ланчи пилота (1.14 млн, cost = 0) раздувают знаменатель
   и ЗАНИЖАЮТ фудкост кухни;
2. дырка в `_unit_sums`: комплимент (paid = 0, cost > 0) сейчас
   добавляет себестоимость юниту без выручки — фудкост ЗАВЫШЕН.
   Продажный фильтр paid > 0 должен стоять и здесь.

**Контракт (аддитивно):** `units` получает `costedRevenue` /
`prevCostedRevenue` — выручка строк, участвующих в фудкосте;
фронт: fc = cost / costedRevenue. Бонус: costedRevenue / revenue =
покрытие техкартами -> новый пункт аудита («доля выручки без
себестоимости > N% — заведите техкарты»).

**Тесты:** юнит с бизнес-ланчем (cost=0) — фудкост не размыт;
комплимент не добавляет cost юниту; покрытие считается.

## 4. Аналитика комплиментов (дизайн принят)

**Статус:** дизайн принят 13.07.2026 (план Егора). Форма выдачи решена:
блок losses страницы foodcost (фаза 1 уже отдаёт общий котёл
compliments{cost, priceValue, qty} — paid = 0, price > 0).

**Последовательность детализации по причинам:**
1. стандарт причин в паспорт онбординга — канонический список типов
   («Комплимент гостю», «Представительские», «Проработка», «Бракераж»…),
   держится на конвенции имён, как staff-маркеры;
2. изменения в iiko ресторанов при онбординге — завести/переименовать
   типы оплат и скидок по стандарту;
3. поля в выгрузку (№6: DishType, ItemSaleEventDiscountType,
   NonCashPaymentType) + колонки + перезаливка;
4. только теперь код: _compliment_sums получает GROUP BY по причине,
   схема Compliments — аддитивную разбивку byReason
   [{reason, cost, priceValue, qty}]; старые поля остаются.

**Ловушка критерия:** канал «именная скидка ~99%» (ресторан) сейчас
НЕ в compliments — у таких строк paid > 0 (копеечный), они сидят
в продажах и в discounts. При детализации критерий комплимента
расширить: paid = 0 ИЛИ discount_type в каноничных причинах.
До кода НЕ менять (правило: не писать код под данные, которых нет).

Потери через кассу — три канала (матрица двух заведений, 11.07.2026):
- закрытие без оплаты: `paid_sum = 0, price > 0`; причина —
  в `NonCashPaymentType` («Представительские» 192 тыс. у кофейни);
- именованная скидка ~100%: `ItemSaleEventDiscountType`
  («Скидка 99%» у ресторана — его представительские; сейчас эти
  строки сидят в продажах с копеечной оплатой!);
- Comment — НЕ детектор (недетерминирован), в выгрузку не берём.

Поля `NonCashPaymentType` и `ItemSaleEventDiscountType` — добавить
в выгрузку одной правкой с `DishType` (№6). Скидки при этом остаются
легитимными: блок discounts фудкоста показывает их «+ и −».

У строки-потери три числа: price (упущенная выручка), cost (РЕАЛЬНЫЕ
затраты заведения), qty. Направление дизайна:

- отчёт «гостеприимство/внутренние обороты» (вместе со staff из №1):
  сумма/динамика/топ блюд, доля от выручки;
- cost комплиментов — слагаемое БУДУЩЕГО грязного фудкоста
  (страница foodcost: чистый + комплименты + стафф + списания);
- форма выдачи (эндпоинт? блок дашборда?) — решить при
  проектировании страницы foodcost, отдельно не строить.

## 5. Ручные исключения позиций из отчётов (исследование)

**Статус:** потребность зафиксирована (10.07.2026), дизайн-набросок ниже,
реализация — вместе с модулем настроек инстанса (валюта, график работы —
очередь кандидатов растёт, модуль назрел).

**Зачем.** Автоправила (paid > 0, cost > 0, staff, тройной ноль) закрывают
типовые случаи — бизнес-ланчи из фудкоста выпадают правилом №3 сами.
Ручной список — для позиций, где автоматика бессильна: мерч (у кофейни
свечки в «продажах еды» на 10.5 тыс.), кейтеринг-разовости, тестовые
позиции с настоящими техкартами и выручкой.

**Набросок:**
- таблица `report_exclusions` в БД инстанса (данные, не код):
  `match_type (group|category|name) + value + scope (sales|foodcost|all)`;
  матчинг терпим к регистру/пробелам, как resolve_unit;
- применение НА ЧТЕНИИ (не ingest): список — живая настройка,
  изменение действует со следующего запроса без перезаливки;
- управление: до админки — CLI (`python -m src.cli.exclusions add ...`);
- прозрачность обязательна: отчёты/аудит показывают «N позиций скрыто
  настройками» — молчаливые исключения = «почему не бьётся с кассой».

## 8. Метрика покрытия техкартами → достоверность фудкоста

**Статус:** решение принято 13.07.2026 при работе над страницей фудкоста
(живой кейс: у части блюд категории нет себестоимости — процент без
пометки вводит в заблуждение). Три уровня:

- **(а) фронт, данные уже есть:** подпись достоверности у каждого
  процента фудкоста (тоталы/юниты/группы) — «рассчитан по N% выручки»
  из revenueWithCost/revenue; при 100% скрывается, ниже порога желтеет;
- **(б) фронт:** покрытие разреза < критического порога (~70%,
  уточнить) → вместо процента «—» с тултипом (прецедент §1а:
  fc позиции с cost=0 показываем «—»);
- **(в) бэк, аддитивно к контракту №9:** блок products (name, unit,
  group, qty, revenue, cost) — закрывает сразу ДВЕ хотелки:
  «топ позиций по вкладу в cost» (кейс лимонадов 41%: два вкуса
  с fc 64% и 77%) и «непокрытые позиции» (cost=0 + их выручка —
  рабочий список технологу «заведи техкарты»). Вопрос N (топ или все)
  и место блока — при реализации.

(а) и (б) — в handoff §1б фронтендеру; (в) — бэку после коммита фазы 1.

«Процент техкарт с костами / без» — два возможных ракурса:
- **по выручке**: `revenueWithCost / revenue` — «фудкост рассчитан
  по N% выручки». Данные уже будут в контракте фудкоста (фаза 1) —
  фронт может показать подписью без отдельной работы бэка;
- **по позициям**: доля позиций меню с cost > 0 — «техкарты заведены
  у M из K блюд» + список непокрытых (готовый рабочий список
  для калькулятора).

Связанные места: карточка №3 (поле revenueWithCost), №6-в (порог
покрытия в аудите), страница фудкоста (подпись достоверности).
Если решим не делать отдельной метрикой — покрытие всё равно
косвенно доступно из пары revenue / revenueWithCost.

## 10. Переезд на Postgres (решение принято 13.07.2026)

**Контекст:** после мультиарендного апдейта все рестораны живут в одной
базе (restaurant_id в orders). Узкое место — SQLite: один писатель
на всю базу, синк одного ресторана блокирует запись остальных.
Решение: единая база остаётся, СУБД меняется на Postgres (MVCC —
писатели не блокируют читателей).

**Важно про async:** конкурентную запись решает Postgres, НЕ async.
FastAPI уже конкурентен (sync-эндпоинты в тредпуле). Полный переход
на async-стек (asyncpg + AsyncSession + async-клиент iiko) — отдельная
осознанная миграция «на вырост», не делать её походя вместе с СУБД.

**Состав работ:**

## 13. Страница «Склад» (дизайн в работе, 13.07.2026)

**Источник:** НЕ OLAP SALES. Остатки — /resto/api/v2/reports/balance/stores
(баланс на любую дату: {store, product, amount, sum}); справочники —
corporation/stores (склады, XML), v2/entities/products/list (1858 позиций
у Ташкента: GOODS/PREPARED/DISH/...; parent -> категория), + единицы
измерения (разведать). Скретч: scripts/scratch_warehouse_recon.py.

**Разведка Ташкента (13.07):** 4 склада — Бар 183.6М, Кухня 138.2М,
Хоз.товары 19.9М, Посуда/Инвентарь 0.07М; 621 позиция; 84 позиции
с ОТРИЦАТЕЛЬНЫМ остатком. Вывод: склады у каждого ресторана свои —
мокапный маппинг на k/b/w мёртв, отдаём живым разрезом (имя + стоимость).

**Решения (Егор, 13.07.2026):**
1. непищевые склады НЕ отдаём на страницу, но ХРАНИМ (внедрение позже =
   новый блок контракта, история уже накоплена). Переходный фильтр —
   маркеры имён складов ('хоз', 'посуда', 'инвентарь', 'упаковка',
   'тара') в domain-константах + паспорт; целевое — флаг склада
   в модуле настроек инстанса (очередь кандидатов — см. №5);
2. отрицательные остатки: в тотал НЕ суммируются (занижают стоимость
   склада). Отдельный блок negativeStock (кол-во позиций, сумма дыры,
   топ виновников) + пункт аудита онбординга «чините складской учёт»;
3. архитектура: ПОЛНЫЙ позиционный снапшот раз в день (конец закрытого
   дня) в БД; имена/единицы/категории денормализуются при синке (как
   dish_sales). Хранимых агрегатов НЕТ: динамика = SUM по (restaurant,
   date, store) на чтении; неделя/месяц — режим отображения на фронте
   (прореживание точек). Объём: ~227 тыс. строк/год/ресторан ≈
   35–45 МБ — приемлемо; ретеншн (старше 2 лет — конец недели) —
   страховка на потом, не работа;
4. бэкфил истории при онбординге — balance на прошлые даты серией
   запросов (глубину и нагрузку на iiko разведать).

**Следствие:** второй доменный загрузчик -> оркестратор синка
(предсказан в №9/фаза 2 для writeoffs; склад пришёл первым).
Кнопка синка в Settings = «все домены ресторана».

**Разведка, раунд 2 (закрыто):** история балансов глубокая — январь-2026
и август-2025 отвечают (бэкфил года = ~365 запросов по ~600 строк;
пересчёт на стороне iiko не бесплатен — чанками, не в часы пик).
Единиц измерения 6 (л/кг/порц/шт/уп/м) — тянуть при синке. Папок
номенклатуры 87 (корневых 14), непищевое видно на уровне корневых
папок ПРОДУКТОВ (УПАКОВКА/ХОЗКА/ПОСУДА) — фильтр «не еда» вести
по ОБОИМ признакам (склад-исключение ИЛИ папка-исключение);
категория позиции = ближайшая папка продукта (как group в продажах).

**Решения, раунд 2 (Егор, 15.07.2026), заменяют часть раунда 1:**
1. храним ТОЛЬКО склады, замапленные на k/b/w (не «всё») —
   **маппинг при онбординге**: таблица restaurant_stores
   (restaurant_id, store_id, store_name, unit); незамапленные склады
   не синкаются. До UI — CLI с автоподсказкой по маркерам имён
   (кухня->k, бар->b, вино/алко->w), админ подтверждает;
2. ежедневный ПОЗИЦИОННЫЙ слепок на КОНЕЦ ЗАКРЫТОГО ДНЯ (вчера 23:59);
   повторный синк дня перезаписывает строки (replace);
3. бэкфил истории при онбординге (balance на прошлые даты);
4. **ретеншн: дневные слепки — скользящий год; старше года — только
   конец недели (воскресенье)**, прореживание при синке;
5. отрицательные остатки: решение раунда 1 в силе (в totals не входят,
   отдельный блок negativeStock);
6. (15.07, вечер) **стандарт требует склады зеркально продажам**:
   Кухня/Бар/Вино — паспорт §9 (перенос остатков ~час работы ресторана).
   «Вино в баре» — нарушение переходного периода, ловится аудитом
   (добавить проверку в №6: склады по стандарту). Оговорка про сырьё
   на разных складах — в паспорте, handoff §0 и Swagger.

**Разведка складов трёх ресторанов (15.07):** «Вино» нет НИ У КОГО

юнит w честно нулевой). «Основной склад» Бишкека — ХОЗКА (уголь,

**Контракт и модель:** черновик согласован в чате 15.07 — schemas
Warehouse{asOf, dataBounds, totals[StoreValue], positions[StockPosition],
