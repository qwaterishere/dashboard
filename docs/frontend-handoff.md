## 0. Терминология: unit / group / category (канон)

Три разреза номенклатуры, единые для всех страниц, контрактов и кода:

| Термин | Что это | Источник в iiko | Значения |
|---|---|---|---|
| `unit` | Направление: папка 1-го уровня дерева номенклатуры | `DishGroup.TopParent` | ключи `k` Кухня, `b` Бар, `w` Вино, `o` «вне подразделений» (всё вне трёх папок) |
| `group` | Папка, в которой лежит блюдо (любая глубина вложенности) | `DishGroup` | живой справочник ресторана («Супы», «Коктейли»…) |
| `category` | Категория блюда: атрибут-справочник, НЕ привязан к дереву папок | `DishCategory` | живой справочник; бывает полузаброшен («Без категории») |

Иерархия: блюдо лежит ровно в одной `group`, `group` — ровно в одном
`unit` → юнит = сумма своих групп. `category` — параллельный разрез
поверх дерева. Легаси-имена `cat` (= unit) и `sub` (= category)
уходят с миграцией страниц на v2.

Нюанс страницы «Склад»: там `unit` (k/b/w) — это **физический склад**,
а не папка номенклатуры; юнит склада резолвится ПО КАНОНИЧЕСКОМУ ИМЕНИ
(«Кухня»/«Бар»/«Вино», паспорт §9 — стандарт именования и есть
настройка, таблицы маппинга нет). Семантика ключей общая с продажами,
но в деньгах складской юнит с продажным не сходится: сырьё одного блюда
хранится на разных складах (мука соуса — «Кухня», вино соуса — «Вино»).
Это не баг, а два разных вопроса: «где лежит» и «что продаём».

## 1. Контракт v2 страницы «Дашборд» — `GET /api/dashboard`

**Статус: ДЕЙСТВУЕТ (04.07.2026), фронт мигрировал.** Версионирование

Страница живёт на v2 (проверено вживую при UX-аудите 17.07.2026:
KPI, LfL, план/прогноз, график, фудкост-блок).

Контракт проверен сквозным тестом 07.07.2026: временный адаптер v2->v1
поверх текущего рендера отрисовал страницу на реальных данных без ошибок
в консоли (KPI, LfL-бейджи, график, поповеры). Референс-реализация всех
производных (LfL, доли, фудкост юнитов, тексты попапов) —
`docs/mockups/dashboard-v2-adapter.reference.js`: можно взять формулы
оттуда, отображение — на ваше усмотрение.

**Принцип v2:** бэкенд отдаёт факты (числа, даты числами), фронтенд —
всё представление: форматирование сумм и валюту, названия месяцев
и падежи, проценты LfL и стрелки, доли юнитов, подписи. Прецедент —
страница sales (фронт уже считает rev/gp/fc из qty/price/unitCost).

### Пример ответа (фикстура)

```json
{
  "period":  {"year": 2026, "month": 7, "dayFrom": 1, "dayTo": 2},
  "compare": {"year": 2025, "month": 7, "dayFrom": 1, "dayTo": 2},

  "kpis": {
    "revenue":  {"value": 590572, "prevValue": 620013, "forecast": null},
    "checks":   {"value": 91,     "prevValue": 102,    "forecast": null},
    "guests":   {"value": 210,    "prevValue": 260,    "forecast": null},
    "avgCheck": {"value": 6490,   "prevValue": 6078,   "forecast": null}
  },

  "revenueByDay": [
    {"day": 1, "weekday": 3, "revenue": 244044, "checks": 47,
     "guests": 130, "plan": null},
    {"day": 2, "weekday": 4, "revenue": 346528, "checks": 44,
     "guests": 80,  "plan": null}
  ],

  "units": [
    {"key": "k", "revenue": 0,       "cost": 0,     "prevRevenue": 0,      "prevCost": 0},
    {"key": "b", "revenue": 0,       "cost": 0,     "prevRevenue": 0,      "prevCost": 0},
    {"key": "w", "revenue": 305916,  "cost": 99325, "prevRevenue": 287100, "prevCost": 90200},
    {"key": "o", "revenue": 8863748, "cost": 0,     "prevRevenue": 8120000,"prevCost": 0}
  ],

  "reviews": null,
  "stock": null
}
```

### Семантика полей

- `period` — показываемый период: месяц последнего закрытого дня,
  с 1-го числа (`dayFrom`) по последний закрытый день (`dayTo`).
- `compare` — период сравнения: те же календарные числа прошлого года
  (29 февраля в невисокосном году усекается до 28-го).
- `kpis.*.prevValue` — значение метрики за `compare`; LfL-процент и стрелку
  фронт считает сам: `(value - prevValue) / prevValue`.
- `checks`/`guests` — только по чекам **с выручкой**: целиком бесплатный
  чек (представительские, стафф-питание) — не продажа и не размывает
  средний чек. Гости платного чека с бесплатными позициями считаются все.
- `kpis.avgCheck` = выручка/чеки, посчитан бэком для удобства
  (в т.ч. прогнозный = forecast выручки / forecast чеков).
- `forecast` — run-rate-прогноз на конец месяца (month-mode) или на
  конец года (year-mode: `?year=YYYY` без `month`) по средним РАБОЧИМ
  дням недели; регулярные выходные распознаются по 8 неделям истории
  и прогнозируются нулём, разовый простой прогноз не занижает.
- `revenueByDay` — полный календарь периода, закрытые дни заведения
  присутствуют с нулями. `weekday`: 0=вс..6=сб (совместим с
  `WEEKDAYS_SHORT` в palette.js).
- `units` — суммы по подразделениям. `key`: k/b/w + `o` («вне
  подразделений» — группы iiko вне папок Кухня/Бар/Вино). Доли
  и фудкост-проценты — производные на фронте: доля = revenue юнита /
  сумма revenue k+b+w (`o` в донат не входит, в общую выручку входит);
  фудкост = cost/revenue; дельта к прошлому году — из prev*-полей.

### Семантика null и нуля (важно!)

| Значение | Смысл | Отображение (предложение) |
|---|---|---|
| `null` | **модуля/данных ещё нет** (планы не внесены, отзывы не подключены, прогноз < 7 закрытых дней) | скрыть блок или «—» |
| `0` | **честный ноль из данных** (день закрыт, юнит без продаж, гости не введены) | показать 0 — это сигнал «проверь кассу» |

- `plan: null` — таблицы целей нет (появится с модулем targets).
- `reviews: null`, `stock: null` — до модулей отзывов и склада;
  их форма будет согласована при появлении модулей.
- `forecast: null` — в месяце меньше 7 закрытых дней.

### Что умерло из v1 (не ищите в v2)

`greeting`, `period.label/note/compareWith` (строки), `kpis.*.lfl`,
`perGuest/perCheck`, `qualityFlag`, `revenueByDay[].avg`,
`revenueByDayMax`, `categories`, `foodcostMini`, `details` —
всё это производные или представление, теперь зона фронта.

## 1а. Контракт v2 страницы «Продажи» — `GET /api/sales` (спецификация)

**Статус:** спроектирован (10.07.2026), реализация начата тактами
(docs/backend-todo.md №7).

**Такт 1 — ДЕЙСТВУЕТ (17.07.2026): суммы-факты уже отдаются.**
`positions[].revenue` (Σ фактических оплат) и `positions[].cost`
(Σ себестоимости; 0 = техкарты нет) добавлены аддитивно; `price` и
`unitCost` — LEGACY-мост, живут до миграции фронта на суммы, затем
удаляются согласованным коммитом. Просьба: перевести computeSalesRaw
на суммы (сейчас rev восстанавливается как qty*price с потерей копеек).

Формулы производных на фронте (из qty/revenue/cost):
цена порции = revenue/qty; себестоимость порции = cost/qty;
gp = revenue − cost; fc = cost/revenue; доля = revenue/Σrevenue;
итоги и донат — суммированием ФАКТОВ (никогда price*qty);
при cost = 0 — fc и gp показывать «—» (правило ниже уже действует).

Известный нюанс данных: cost позиции включает нули строк без техкарт —
у позиции, покрытой техкартой не весь период, cost/qty занижен
(решение — в следующих тактах, кандидат: правило №3 как в фудкосте).

**Такт 2 — ДЕЙСТВУЕТ (17.07.2026): дефолт и усечение периода.**
Без параметров отдаётся месяц последнего закрытого дня (раньше — ВСЯ
история: «ложь периода» из UX-аудита). Явные date_from/date_to
усекаются краями данных, эффективные границы видны в period.label;
одна дата без второй — 422; период вне данных — пустая страница
с label «Нет данных за период».

⚠️ **Обязательная парная правка фронта:** слать date_from/date_to
ВО ВСЕХ режимах (сейчас sales-data.store.ts шлёт только в week:
`useRange = granularity() === 'week'` — убрать условие). Дефолт бэка —
страховка, канон — явные даты. До правки режим «Год» показывает
текущий месяц под годовой подписью.

**listValue — ДЕЙСТВУЕТ (17.07.2026):** `positions[].listValue` —
прейскурантная выручка (Σ цен меню без скидок): цена меню =
`listValue/qty`, скидки позиции = `listValue − revenue` (готовый
позиционный разрез скидок — колонка/тултип по желанию).

**Такт 4 — ДЕЙСТВУЕТ (17.07.2026): факты периода в ответе.**
`period.dateFrom/dateTo` (ISO) — ФАКТИЧЕСКИ покрытый период после
дефолта и усечения (запросили по 31-е, закрыто по 16-е -> в ответе
16-е); null — данных нет. `label/note` — LEGACY, умирают в такте 5.

Остальные такты (в очереди): id-склейка переименований (dish_id уже
в БД, ждёт пересоздания базы); rename sub/cat -> category/unit
+ добавление group + снос легаси (такт 5, ломающий — один
согласованный коммит с миграцией фронта); решение по «разбавленному»
cost (правило №3).

### Чек-лист миграции фронта на sales v2 (сводный, 17.07.2026)

Можно делать уже сейчас, по одному пункту, всё аддитивно:

1. **computeSalesRaw на суммы**: rev/cost брать из `revenue`/`cost`
   ответа, а не восстанавливать qty*price (копейки). Итоги/донат/ABC —
   только суммированием фактов;
2. **слать date_from/date_to во всех режимах** (sales-data.store.ts:
   убрать условие `granularity() === 'week'`) — иначе «Год» показывает
   месяц;
3. **подпись периода — из period.dateFrom/dateTo** (свой формат
   и локаль), label больше не читать;
4. **режимы страницы (решение продукта 17.07)**: день / неделя / месяц
   + произвольный диапазон дат в календаре; пресет «Год» убирается
   (годовой срез достижим диапазоном). Период-бар становится
   пер-страничным (складу нужен день-пикер — та же работа);
5. (опционально) бары диаграммы и строки таблиц трекать по `id`
   позиции, где он есть.

**Такт 5 (по готовности пп. 1–3 — согласовать дату):** бэк
переименовывает sub -> category, cat -> unit, добавляет group,
удаляет price/unitCost и period.label/note; фронт синхронно меняет
маппер. До этого никакие поля не исчезают.

**Принцип тот же:** факты-суммы, производные — фронт.

```json
{
  "period": {"year": 2026, "month": 7, "dayFrom": 1, "dayTo": 8},
  "positions": [
    {"name": "Цезарь с цыпленком", "unit": "k", "group": "Салаты",
     "category": "Салаты", "qty": 144.0, "revenue": 102319, "cost": 29384}
  ]
}
```

- Дефолтный период — **текущий месяц** (месяц последнего закрытого дня),
  как у дашборда; `?date_from/date_to` опциональны, усечение последним
  закрытым днём, эффективные границы в ответе;
- `positions` — только продажные строки (paid > 0; staff и техшум
  исключены на бэке после пакета правил учёта);
- фронт считает: price = revenue/qty, gp = revenue − cost,
  fc = cost/revenue, ABC, донат = sum(revenue) по unit
  ('o' в донат не входит), подкатегории = по group;
- **`cost = 0` = техкарты нет**: позиционные fc и gp показывать «—»
  (числа 0%/полная маржа — враньё), позиция в ABC участвует по выручке;
- `category` может быть менее осмысленной, чем `group` (у некоторых
  ресторанов категории полузаброшены) — переключатель разреза желателен;
- что умерло из текущего: `period.label/note`, `sub`,
  средние `price`/`unitCost` (замена: суммы `revenue`/`cost`);
  `cat` переименован в `unit`;
- зарезервировано на будущее (аддитивно): `isModifier` (появится
  вместе с DishType в выгрузке — метрика допродаж), `prevRevenue`
  (LfL позиций — по запросу фронта, не сейчас).

## 1б. Фудкост: блок products для диаграммы «Выгодность позиций» (16.07.2026)

`GET /api/foodcost` отдаёт `products`. Фронт мапит в view-model
(`buildProducts` в `foodcost.mapper.ts`): цена/себестоимость порции =
revenue/qty и cost/qty; диаграмма «Выгодность позиций» рендерит топ-10/
худшие-10 по fc%.

```json
"products": [
  {"id": "92113b71-0481-4d5e-97f2-68254027655c",
   "name": "Феттучини альфредо", "unit": "k", "qty": 129.0,
   "revenue": 14080000, "listValue": 15480000, "cost": 4397435}
]
```

- **`listValue` (17.07, решение Егора): «Цена» в тултипе — МЕНЮШНАЯ** =
  `listValue / qty` (прейскурант, круглая; сейчас показывается средняя
  фактическая `revenue/qty` — некруглые 26 679 вместо 45 000).
  Скидки позиции = `listValue − revenue` (рекомендуем третьей строкой
  тултипа — объясняет разницу). **fc считать ПО-ПРЕЖНЕМУ от `revenue`**
  (фактическая экономика; от прейскуранта был бы приукрашен);

- `id` — UUID блюда в iiko (стабильная идентичность, как productId
  на складе): рекомендуем трекать бары диаграммы по нему, не по имени;
  `name` — актуальное имя из последней продажи периода: переименованное
  блюдо НЕ двоится (склейка по id на бэке). `id: null` возможен только
  у исторических строк до пересоздания базы;
- то же лечение у `groups`: папка, переименованная в середине периода,
  отдаётся одной строкой под последним именем, LfL групп переживает
  переименование (матч prev-периода по id папки).

- ВСЕ позиции с техкартами за period (правило №3: paid > 0 и cost > 0),
  по убыванию выручки; позиции без техкарт исключены на бэке (fc 0%
  возглавил бы рейтинг «лучших»); `unit` включает `'o'`;
- фронт считает: цена порции = revenue/qty, себестоимость порции =
  cost/qty, fc = cost/revenue; топ-10/10 — как сейчас в утилях;
- **предложение — порог шума**: на живых данных позиция с qty=1
  попадает в «худшие» со случайной ценой (реальный кейс: вино по
  бокалу, продано 1 раз, fc 71%). Рекомендуем минимальный qty (или
  мин. выручку) для участия в рейтинге — границу подберём вместе
  (research у бэка);
- известный шум: стафф-позиции по спеццене («Капучино стафф», fc ~60%)
  будут в рейтинге, пока не решена staff-механика (карточка №9) —
  это честные данные, не баг.

## 2. Страница «Склад» v2: 404 и day-picker (16.07.2026)

**Статус: ДЕЙСТВУЕТ.** `GET /api/warehouse` отдаёт контракт v2 из БД
(схема `Warehouse` в Swagger).

1. **Эндпоинт может отвечать 404** в двух случаях:
   - слепков ещё нет вообще (новый ресторан до первого синка склада) —
     страница должна иметь состояние «данных нет»;
   - `?date` вне `dataBounds.availableDates`.
   При корректном day-picker'е второй случай не наступает: пикер обязан
   гасить все даты вне `availableDates` (список кодирует и ретеншн —
   глубже скользящего года только воскресенья, — и дыры синка).
2. **`positions` содержит и минусовые строки** (`qty < 0` — грязный
   учёт), а `totals` и точки `dynamics` считают ТОЛЬКО положительные:
   сумма positions ≠ totals — это не баг. Готовое предупреждение
   о минусах — блок `negativeStock`.
3. **`dynamics` — единственный источник графика** «стоимость склада
   во времени»; неделя/месяц — прореживание точек на фронте. День без
   слепка в точках отсутствует (дыра в линии, не ноль).
4. **(17.07, из UX-аудита) Странице нужен day-picker, а не общий
   период-бар**: сейчас сверху «Неделя/Месяц/Год + LfL», а контракт —
   слепок на конкретный день (`?date` из `dataBounds.availableDates`,
   пикер гасит остальные даты). Период-бар становится пер-страничным —
   та же работа, что режимы продаж (чек-лист §1а, п. 4).

## Общие заметки по контрактам

1. **`/api/sales`: `positions[].cat` имеет четвёртое значение `'o'`**
   («вне подразделений»). В донат юнитов не входит; в таблице позиций,
   ABC и общей выручке участвует (в Angular `CAT_NAME` уже покрывает).
2. **`/api/sales`: `positions[].qty` — дробный** (вес: 4.5 кг).
3. **Валюта: «₽» захардкожен в шаред-форматировании** (FmtPipe) —
   у пилотов сумы/сомы; аудит 17.07 подтвердил «₽» на живых страницах.
   Нужна настройка валюты инстанса (конфиг фронта).
4. **`lfl` в v1-контрактах остальных страниц**: у ресторана без прошлого
   года сравнивать не с чем — при миграции страниц на v2-стиль
   закладывайте `prevValue: null`.

2. **Два языка процентов на фудкосте**: «31,8 %» (доля), «+22,6 %»
   (отклонение ОТ ЦЕЛИ, относительное) и «LfL +14,2 %» в одном экране
   неразличимы. Предложение: отклонение от цели — в п.п. (на дашборде
   уже так: «+1,8 п.п.»), LfL — только бейджем единого стиля.
3. **Рейтинг «Выгодность позиций»**: топ лучших забит копеечными
   (мята/чай/багет fc 1–5%) — нужен порог шума (см. §1б, границу
   подберём вместе); стафф-позиции (fc ~62%) в худших — честные данные
   до staff-механики, но можно скрыть представлением. Дубли позиций
   (Мята x2) — временное состояние базы, уйдёт после пересоздания
   с бэкфиллом (бэк).
4. **ABC на продажах — плотность**: 535 строк лентой по ~60px; нужны
   поиск, свёртка C-класса (278 строк = 5% прибыли), компактный режим,
   сокращение сумм («714,0 млн» вместо 11 знаков).
5. **Дашборд-сводка**: карточки дублируют факты (чеки/гости/средний
